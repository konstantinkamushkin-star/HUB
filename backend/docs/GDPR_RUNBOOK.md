# GDPR Runbook (операционный минимум)

Этот документ фиксирует рабочий процесс команды по GDPR/DSAR для DiveHub.
Он дополняет политику конфиденциальности и технические endpoint'ы.

## 1) Область действия

- Backend: `backend/src`
- iOS: `DiveHub`
- Android: `DiveHubAndroid`
- Admin web: `admin-web`

## 2) Роли и ответственность

- **Owner (Data Privacy Lead):** принимает финальные решения по DSAR.
- **Backend engineer:** выполняет export/delete, проверяет аудит и логи.
- **Support/admin:** принимает обращение, делает первичную верификацию личности.
- **Security/DevOps:** участвует в инцидентах и расследованиях утечек.

## 3) Типы запросов субъекта данных (DSAR)

- Access (экспорт данных)
- Erasure (удаление/анонимизация)
- Rectification (исправление)
- Restriction/Objection (ограничение/возражение)

## 4) SLA и дедлайны

- Подтверждение получения: до 72 часов.
- Полный ответ по запросу: до 30 календарных дней.
- Продление (если сложно): +30 дней с уведомлением пользователя и причиной.

## 5) Каналы приема запросов

- Email из privacy policy.
- Встроенные endpoint'ы пользователя:
  - `GET /api/users/me/export`
  - `DELETE /api/users/me`
- Внутренний admin-flow для ручной обработки нестандартных кейсов.

## 6) Процесс обработки DSAR

1. **Регистрация запроса**
   - Зафиксировать дату, канал, user id/email, тип запроса.
   - Присвоить ticket id.
2. **Верификация личности**
   - Если запрос из авторизованной сессии — проверить валидность токена.
   - Если по email — запросить подтверждение владения аккаунтом.
3. **Оценка правового основания**
   - Проверить, нет ли обязательств хранения по закону (финансы, споры, fraud).
4. **Выполнение**
   - Export: отдать структурированный payload в безопасном канале.
   - Delete: запустить анонимизацию/удаление согласно текущему коду.
5. **Подтверждение пользователю**
   - Сообщить результат и дату завершения.
6. **Аудит**
   - Убедиться, что события зафиксированы в audit logs.

## 7) Чеклист по запросу на экспорт

- [ ] Идентичность подтверждена.
- [ ] Экспорт сформирован через `GET /api/users/me/export` или админ-инструмент.
- [ ] Проверено, что секретные поля не выданы (пароли, totp secret, reset codes).
- [ ] Ответ отправлен пользователю безопасным каналом.
- [ ] Событие зафиксировано в аудите.

## 8) Чеклист по запросу на удаление

- [ ] Идентичность подтверждена.
- [ ] Пройдена re-auth проверка (пароль или recent session).
- [ ] Выполнен `DELETE /api/users/me`.
- [ ] Проверено, что аккаунт переведен в `deleted` и PII анонимизированы.
- [ ] Проверено, что user-generated data обезличены по текущей логике.
- [ ] Пользователь уведомлен о завершении.
- [ ] Событие зафиксировано в аудите.

## 9) Исключения и отказ

Запрос может быть отклонен/ограничен, если:

- нет подтверждения личности;
- есть обязательства хранения по закону;
- запрос явно злоупотребительный/повторный без новых оснований.

При отказе обязателен письменный ответ с причиной.

## 10) Что проверять ежемесячно

- Работают ли `me/export` и `me` delete в production.
- Есть ли audit trail для user GDPR действий.
- Нет ли регрессий по минимизации PII в публичных endpoint'ах.
- Обновлены ли privacy policy и список subprocessors.
