# Подготовка к production (до аренды сервера)

Чеклист и решения, которые стоит зафиксировать **до** выбора VPS. Реальные секреты храните в менеджере паролей или в секретах CI/CD — **не** коммитьте в репозиторий.

## 1. Секреты и переменные окружения

| Переменная | Назначение | Где хранить значение |
|------------|------------|----------------------|
| `JWT_SECRET` | Подпись JWT | Менеджер паролей; сгенерировать: `openssl rand -base64 48` |
| `DB_*` | PostgreSQL | Секреты хостинга / managed DB |
| `REDIS_PASSWORD` | Redis (если включён) | Секреты хостинга |
| `SMTP_*` | Почта (сброс пароля и т.д.) | Секреты |
| `OPENAI_API_KEY` | Импорт поездок (опционально) | Секреты |

Полный перечень см. в корне backend: `.env.example`.

## 2. Домены и HTTPS

Запишите заранее (в этом файле или в wiki), без реальных секретов:

- **API**: один поддомен удобен для TLS и CORS, например `https://api.divehub.example`.
- **CORS_ORIGINS**: только точные origin’ы браузерных клиентов (с протоколом и без trailing slash):
  - админка: `https://admin.divehub.example`
  - веб-приложение (если есть): `https://app.divehub.example`
- **Нативное iOS-приложение** обычно не шлёт браузерный `Origin`; для мобильного API часто достаточно тех origin’ов, которые реально открывают API в браузере (админка, веб). Уточните под ваши клиенты.

HTTPS на границе: **Caddy** или **Nginx** + Let’s Encrypt, либо TLS у облачного балансировщика.

## 3. Проверка «как в проде» локально

1. PostgreSQL **с PostGIS**, пустая или тестовая БД.
2. Прогнать миграции (как у вас принято: скрипты `npm run migrate:*` из `package.json`).
3. В `.env` для проверки:
   - `NODE_ENV=production`
   - `CORS_ORIGINS=https://ваш-тестовый-origin.example` (или список через запятую) — **обязательно**, иначе приложение не стартует в production.
4. Команды:

```bash
npm ci
npm run verify:prod-build   # только сборка
# затем с поднятой БД и Redis:
npm run start:prod
```

Проверка: `GET /api/health`, при необходимости — авторизация и критичные эндпоинты.

**Docker (опционально):** из папки `backend` — `docker compose up --build` (см. `docker-compose.yml`). После первого старта БД выполните миграции с хоста или из контейнера API (те же `npm run migrate:*` при установленных зависимостях) — автоматический прогон миграций в compose не встроен.

## 4. Медиа (`uploads/media`)

| Сценарий | Рекомендация |
|----------|----------------|
| Один инстанс API | Достаточно каталога на диске + бэкап тома |
| Несколько инстансов | Общий сетевой том **или** объектное хранилище (S3-совместимое) + доработка `MediaService` |

Текущая реализация — **локальный диск** относительно рабочей директории процесса.

## 5. Где держать PostgreSQL

- **На том же VPS**, что и API: проще старт, вы сами настраиваете бэкапы и обновления; нужен образ/пакет с **PostGIS**.
- **Managed** (Neon, Supabase, RDS, Aiven и т.д.): проще бэкапы и доступность; VPS можно взять меньше по RAM.

От выбора зависит размер VPS и политика бэкапов.

## 6. Redis

В production используйте **отдельный** Redis (контейнер на том же хосте или managed). В коде при недоступности Redis включается in-memory кэш — для продакшена это **не** целевой режим (рассинхрон между рестартами и инстансами).

## 7. WebSocket (чат)

- **Один инстанс** API — проще всего.
- **Несколько инстансов** за балансировщиком — нужны **sticky sessions** (cookie/session affinity) **или** доработка архитектуры (общий pub/sub, другой транспорт).

Решение заранее влияет на конфиг балансировщика и масштабирование.

## 8. За reverse proxy

Когда API за Nginx/Caddy/облаком, задайте `TRUST_PROXY` (см. `.env.example`) — иначе rate limit и логи по IP могут быть неверными.
