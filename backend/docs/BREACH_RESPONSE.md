# Data Breach Response (72h Playbook)

Этот документ описывает действия команды при подозрении на утечку/компрометацию персональных данных.

## 1) Критерии инцидента

Инцидентом считается:

- несанкционированный доступ к персональным данным;
- утечка дампа БД/логов с PII;
- компрометация токенов/учетных данных в production;
- массовое несанкционированное изменение/удаление пользовательских данных.

## 3) Таймлайн действий

### T+0..2 часа (обнаружение и containment)

- [ ] Открыть incident ticket, присвоить severity.
- [ ] Зафиксировать источник сигнала (alert, лог, жалоба).
- [ ] Ограничить ущерб: revoke токены/ключи, ограничить доступ, изолировать узел.
- [ ] Сохранить доказательства (логи, снимки, хэши артефактов).

### T+2..24 часа (оценка масштаба)

- [ ] Определить категории затронутых данных.
- [ ] Оценить количество пользователей и регионов.
- [ ] Определить root cause (или рабочую гипотезу).
- [ ] Принять решение о необходимости уведомления регулятора и пользователей.

### T+24..72 часа (уведомления и план восстановления)

- [ ] Подготовить регуляторное уведомление (если требуется).
- [ ] Подготовить уведомление пользователям (если высокий риск для их прав/свобод).
- [ ] Выпустить технический mitigation и план устранения.
- [ ] Зафиксировать статус и owner'ов corrective actions.

## 4) Минимальный набор данных в отчете инцидента

- Дата/время обнаружения и подтверждения.
- Какие системы затронуты.
- Какие данные затронуты (категории PII).
- Оценка числа затронутых субъектов.
- Что уже сделано для containment.
- Какие дальнейшие шаги и сроки.

## 5) Технический чеклист containment

- [ ] Ротация JWT secret/ключей (если есть риск компрометации).
- [ ] Инвалидация активных сессий/refresh токенов.
- [ ] Блокировка уязвимого endpoint/фичи.
- [ ] Усиление rate limit/WAF правил при атаке.
- [ ] Проверка audit logs на массовые подозрительные операции.

## 6) Коммуникации

- Внешние комментарии дает только назначенный spokesperson.
- Support получает шаблон ответа и FAQ.
- Не публиковать непроверенные причины до завершения расследования.

## 7) Postmortem (обязательно)

Срок: до 5 рабочих дней после стабилизации.

- [ ] Root cause analysis.
- [ ] Что сработало/что не сработало в процессе.
- [ ] Список corrective/preventive actions с дедлайнами.
- [ ] Обновление runbook/алертов/мониторинга.
