# UX-потоки бронирования: дайвцентр и бассейн

Документ описывает единый пользовательский процесс бронирования для двух сценариев:
- `open_water` — бронирование дайвинга в дайвцентре (поездка, группа, несколько дней)
- `pool` — запись в бассейн (сессия, аренда снаряжения, быстрое подтверждение)

Цель: один понятный UX, один backend-процесс, минимальная путаница для пользователя.

---

## 1) Общая логика для обеих веток

Единая модель заявки:

- Черновик формы создается на клиенте.
- После отправки создается `booking` со статусом `pending`.
- Автоматически открывается `chatConversationId` между клиентом и организатором.
- Организатор отвечает одним из действий:
  - `confirm` -> бронь становится `confirmed`
  - `counter_offer` -> бронь становится `quoted`, пользователь должен принять/отклонить
  - `decline` -> бронь становится `cancelled`

Рекомендуемые статусы:

- `draft` (локально на клиенте)
- `pending` (заявка отправлена)
- `quoted` (получены условия от центра/бассейна)
- `confirmed` (подтверждено обеими сторонами)
- `cancelled` (отменено любой стороной)
- `completed` (услуга оказана)

## 2) Кейс 1: поездка в Египет, 5 человек, 4 дня, пожелания по инструктору

### Экран A: карточка дайвцентра

CTA: `Забронировать дайвинг`

Передаем в мастер:
- `bookingType=open_water`
- `diveCenterId`

### Экран B: параметры поездки

Поля:
- Страна/локация (предзаполнено из карточки, редактируемо)
- Дата начала
- Дата окончания
- Количество участников (по умолчанию 1)
- Формат: `fun dive` / `course` / `mixed`

Валидации:
- `dateEnd >= dateStart`
- `participantsCount >= 1`

### Экран C: участники и опыт

Для каждого участника (или только агрегированно на первом этапе):
- Имя
- Уровень сертификации (если есть)
- Нужна аренда снаряжения (да/нет)
- Комментарии по здоровью/ограничениям (опционально)

Быстрый вариант MVP:
- сначала только `participantsCount` и общий флаг аренды
- детализацию участников добирать в чате после отправки заявки

### Экран D: пожелания к инструктору

Поля:
- Предпочитаемый язык инструктора
- Предпочтительный уровень/специализация
- Доп. пожелания (свободный текст)

Пример:
- "Русскоязычный инструктор, опыт с группами новичков"

### Экран E: итог и отправка

Показываем:
- даты
- количество участников
- ключевые пожелания
- предварительная вилка цены (если доступна)

CTA:
- `Отправить заявку`

После отправки:
- статус `pending`
- авто-переход в чат с системным сообщением:
  - "Новая заявка: Египет, 4 дня, 5 участников, пожелания по инструктору указаны."

### Экран F: чат и подтверждение

В чате обе стороны уточняют:
- расписание
- состав группы
- финальную стоимость
- кто и что арендует

Когда центр отправляет оффер:
- статус `quoted`
- клиент видит экран/баннер:
  - условия
  - кнопки `Принять` / `Отклонить`

При принятии:
- статус `confirmed`
- показываем карточку подтвержденной брони

## 3) Кейс 2: запись в бассейн в Москве (с/без аренды)

### Экран A: карточка бассейна

CTA: `Записаться в бассейн`

Передаем:
- `bookingType=pool`
- `diveCenterId` или `poolId` (в зависимости от доменной модели)

### Экран B: слот

Поля:
- дата
- временной слот
- длительность (если применимо)
- количество участников

Если доступно расписание слотов:
- показываем только свободные слоты

### Экран C: снаряжение

Поля:
- `Без аренды` / `С арендой`
- если аренда включена: типы и размеры

MVP:
- один переключатель `Нужна аренда`
- детализация размеров в чате или на экране подтверждения

### Экран D: подтверждение

Показываем:
- слот
- количество участников
- аренда
- итоговая цена

CTA:
- `Подтвердить запись`

Логика подтверждения:
- Если слот полностью управляется расписанием -> instant confirm (`confirmed`)
- Если требуется ручная проверка -> `pending` + чат + оффер/подтверждение

---

## 4) Единые UX-правила

- Один термин в интерфейсе: "Бронирование"
- Один список "Мои бронирования" с фильтрами:
  - `pending`, `quoted`, `confirmed`, `cancelled`, `completed`
- Из карточки брони всегда доступен чат.
- В каждом статусе понятный следующий шаг:
  - `pending`: "Ждем ответ организатора"
  - `quoted`: "Примите или отклоните условия"
  - `confirmed`: "Бронь подтверждена"

## 6) MVP-приоритет внедрения

1. Оставить единый мастер бронирования, добавить выбор `open_water/pool`.
2. Для `open_water` дать multi-day поля + пожелания к инструктору.
3. Для `pool` сделать короткий flow со слотом и арендой.
4. После `createBooking` всегда открывать чат и системное сообщение.
5. Добавить состояние `quoted` и действия `accept/decline` для пользователя.

Это даст рабочий end-to-end без перегруза первого релиза.
